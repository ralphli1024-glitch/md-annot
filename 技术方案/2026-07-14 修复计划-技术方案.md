## 1. 编辑模式：移除高亮、划线和批注后的右上角圆形、三角形和正方形

### 需求理解

编辑模式（CodeMirror 6）中，每条批注文本的右上角会显示一个小标记——高亮是圆形、划线是三角形、批注是正方形。这些标记在批注密集时视觉杂乱，干扰阅读体验。需要完全移除。

### 当前实现分析

在 `cm6-decorations.ts` 中，`highlightDecoration` 和 `commentDecoration` 函数为装饰元素设置了 `--md-annot-indicator-color` CSS 变量。在 `styles.css` 中，`.md-annot-highlight::after`、`.md-annot-underline::after`、`.md-annot-comment::after` 等伪元素使用这个变量渲染形状标记。划线（UNDERLINE）没有对应的伪元素样式，所以没有标记。

### 实现效果

批注文本仅有背景色（高亮）或波浪线（划线），没有任何右上角的几何形状标记。阅读体验更干净。

### 技术方案

**步骤 1：移除 `cm6-decorations.ts` 中的 indicator 变量**

`highlightDecoration` 和 `commentDecoration` 不再设置 `--md-annot-indicator-color` 变量：

```typescript
// 修改前
function highlightDecoration(color = "#90EE90") {
  const dot = darkenColor(color, 0.35);
  return Decoration.mark({
    class: "md-annot-highlight",
    attributes: {
      style: `background-color: ${color}; --md-annot-indicator-color: ${dot};`,
    },
  });
}

// 修改后
function highlightDecoration(color = "#90EE90") {
  return Decoration.mark({
    class: "md-annot-highlight",
    attributes: {
      style: `background-color: ${color};`,
    },
  });
}
```

同理修改 `underlineDecoration` 和 `commentDecoration`。

**步骤 2：移除 `styles.css` 中的伪元素样式**

删除针对 `.md-annot-highlight::after`、`.md-annot-underline::after`、`.md-annot-comment::after` 的全部 CSS 规则。

**步骤 3：清理 `darkenColor` 引用**

如果 `darkenColor` 函数不再被任何装饰函数调用，则一并删除（保留对 `floating-toolbar.ts` 的影响——那里也使用了同样的 `darkenColor`，两者互不影响）。

| 编辑模式创建划线 | 文本有波浪线，右上角无标记 | 目视检查 |
| 编辑模式创建批注 | 文本有背景色，右上角无方块 | 目视检查 |

## 5. 所有操作支持撤销（Cmd+Z / Ctrl+Z）

### 需求理解

当前批注操作（创建、删除、更新）是一次性的，无法撤销。用户需要通过撤销快捷键（macOS: Cmd+Z, Windows/Linux: Ctrl+Z）撤销最近一次批注操作。需要实现一个通用的撤销/重做栈（Undo/Redo Stack），跨平台兼容。

### 当前实现分析

- `AnnotationManager` 执行 CRUD 后直接修改 `annotations` Map，不保留历史状态
- CM6 的编辑（文字修改）已有原生 Undo，但批注操作是独立状态
- 撤销需要支持：创建批注 → 撤销 = 删除；删除批注 → 撤销 = 恢复；修改批注 → 撤销 = 恢复旧值

### 实现效果

- macOS: Cmd+Z, Windows/Linux: Ctrl+Z 撤销最近一次批注操作（使用 e.metaKey || e.ctrlKey 判断修饰键，天然跨平台）
- macOS: Cmd+Shift+Z, Windows/Linux: Ctrl+Shift+Z 恢复最近一次撤销
- 撤销范围包括：创建高亮/划线/批注、删除批注
- 撤销操作不影响 CM6 原生文字编辑的 Undo 栈
- 栈上限约 50 步，防止内存膨胀

### 技术方案

**步骤 1：定义撤销动作类型**

在 `data-models.ts` 中新增类型：

```typescript
export interface UndoAction {
  type: 'annotation:created' | 'annotation:deleted' | 'annotation:updated';
  timestamp: number;
  payload: any; // 保存前一个状态的快照
}
```

**步骤 2：在 `annotation-manager.ts` 中集成 UndoManager**

```typescript
class UndoManager {
  private undoStack: UndoAction[] = [];
  private redoStack: UndoAction[] = [];
  private readonly maxStack = 50;

  push(action: UndoAction): void {
    this.undoStack.push(action);
    if (this.undoStack.length > this.maxStack) {
      this.undoStack.shift();
    }
    this.redoStack = []; // 新操作清空 redo
  }

  undo(): UndoAction | null {
    const action = this.undoStack.pop();
    if (action) this.redoStack.push(action);
    return action;
  }

  redo(): UndoAction | null {
    const action = this.redoStack.pop();
    if (action) this.undoStack.push(action);
    return action;
  }

  clear(): void {
    this.undoStack = [];
    this.redoStack = [];
  }
}
```

**步骤 3：修改 `AnnotationManager` 的 CRUD 方法**

每种修改操作在修改状态前，先将「逆向操作」压入 Undo Stack。

```typescript
// createAnnotation 修改
createAnnotation(...): Annotation | null {
  // ...现有逻辑...
  const annotation = { ... };
  this.annotations.set(annotation.id, annotation);
  
  // 记录撤销动作：删除该批注
  this.undoManager.push({
    type: 'annotation:created',
    timestamp: Date.now(),
    payload: { id: annotation.id },
  });
  
  this.emit("annotation:created", annotation);
  return annotation;
}

// deleteAnnotation 修改
deleteAnnotation(id: string): boolean {
  const oldAnnotation = this.annotations.get(id);
  if (!oldAnnotation) return false;
  
  this.annotations.delete(id);
  
  // 记录撤销动作：恢复该批注
  this.undoManager.push({
    type: 'annotation:deleted',
    timestamp: Date.now(),
    payload: oldAnnotation,
  });
  
  this.emit("annotation:deleted", { id });
  return true;
}
```

**步骤 4：实现撤销/重做方法**

```typescript
undo(): void {
  const action = this.undoManager.undo();
  if (!action) return;

  switch (action.type) {
    case 'annotation:created':
      // 撤销创建 = 删除
      this.annotations.delete(action.payload.id);
      this.emit("annotation:deleted", { id: action.payload.id });
      break;
    case 'annotation:deleted':
      // 撤销删除 = 恢复
      const anno = deserializeAnnotation(action.payload);
      this.annotations.set(anno.id, anno);
      this.emit("annotation:created", anno);
      break;
      break;
  }
  this.emit("undo:executed");
}

redo(): void {
  const action = this.undoManager.redo();
  if (!action) return;
  // 重做 = 重新执行正向操作
  // ... 反向逻辑 ...
  this.emit("redo:executed");
}
```

**步骤 5：注册键盘快捷键**

在 `main.ts` 中监听快捷键（e.metaKey || e.ctrlKey 兼容跨平台）：

```typescript
this.registerDomEvent(document, "keydown", (e: KeyboardEvent) => {
  const isMod = e.metaKey || e.ctrlKey;
  if (isMod && e.key === "z" && !e.shiftKey) {
    e.preventDefault();
    this.annotationManager.undo();
    this.annotationManager.persist();
    this.updateCM6Decorations(currentView);
  }
  if (isMod && e.key === "z" && e.shiftKey) {
    e.preventDefault();
    this.annotationManager.redo();
    this.annotationManager.persist();
    this.updateCM6Decorations(currentView);
  }
});
```

需注意避开 CM6 编辑器的原生 Undo：仅在 CM6 没有焦点或批注操作栈非空时拦截。

**跨平台实现要点**

- 使用 `e.metaKey || e.ctrlKey` 判断修饰键，macOS 的 Command 和 Windows 的 Control 都响应
- `e.key === "z"` 在两种平台下均一致
- Shift+Z 对应重做，`e.shiftKey` 的判断跨平台兼容
- 不需要区分 navigator.platform，统一处理即可

```typescript
setIcon(deleteBtn, "trash-2");
deleteBtn.addEventListener("click", (e) => {
  e.stopPropagation();
  this.annotationManager.deleteAnnotation(anno.id);
  this.annotationManager.persist();
});
```

## 总体实施建议

### 执行顺序

建议按以下顺序实施，每完成一项提交一次：

| 顺序 | 任务 | 理由 |
|------|------|------|
| 1 | 任务 1 — 编辑器尾标移除 | 改动最小，CSS 级别，无风险 |
| 2 | 任务 6 — 批注列表优化 | 纯 UI 改动，不影响核心逻辑 |
| 3 | 任务 2 — 阅读工具栏钢笔+P | UI 改动，独立模块 |
| 4 | 任务 3 — 阅读工具栏批注图标 | UI 改动，独立模块 |
| 5 | 任务 5 — Undo/Redo | 最复杂，涉及核心状态管理，最后实施 |

### 风险点

| 风险 | 等级 | 应对 |
|------|------|------|
| Undo 与 CM6 原生 Undo 冲突 | 高 | 只在 CM6 无焦点或批注栈非空时拦截 Undo 快捷键 |

| 批注列表优化后批量删除导致动画闪烁 | 低 | 使用 100ms 延迟再刷新 |
| Undo 跨 session 持久化 | 低 | Undo 栈仅存于内存，不持久化 |

### 测试策略

- 每个任务的 CSS 改动：浏览器 DevTools 检查样式覆盖
- 批注列表优化：手动操作所有交互路径
- Undo/Redo：覆盖创建/删除/更新三种场景 + 边界（空栈/满栈）
